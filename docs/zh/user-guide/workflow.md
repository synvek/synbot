# 工作流指南

Synbot 支持 **TurboWorkflow**：可持久化、可恢复的多步工作流。你可以通过描述任务创建工作流、可选地提供 JSON 定义、按步执行，并在中断或等待用户输入后恢复。

## 概述

- **触发方式**：消息以 `/workflow` 开头（不区分大小写）。不依赖其他关键词或意图识别。
- **创建**：`/workflow <描述>` 或 `/workflow` 加可选 JSON 定义。机器人会根据描述生成步骤，或使用你提供的 JSON（需确认后执行）。
- **继续**：`/workflow continue` 或 `/resume` 可恢复当前会话中已暂停或中断的工作流。
- **控制**：在工作流（或 agent）运行期间，可使用 `/stop`、`/status`、`/clear`、`/resume`，而不会启动新任务。

## 指令说明

### 工作流指令

| 指令 | 说明 |
|------|------|
| `/workflow <描述>` | 创建并运行新工作流。消息其余部分为任务描述，由模型生成步骤。 |
| `/workflow continue` | 继续当前会话已保存的工作流（如暂停或超时后）。 |
| `/workflow` + JSON | 可在消息中附带 JSON 工作流定义（代码块或 `{ ... }`）。机器人会解析、展示步骤摘要并请你确认后再执行。 |

### 控制指令（忙碌时或随时可用）

| 指令 | 说明 |
|------|------|
| `/stop` | 停止当前正在运行的工作流或任务。 |
| `/resume` | 与“继续工作流”等价，针对当前会话。 |
| `/status` | 查看当前会话信息及工作流状态（若有）：会话 key、消息数、是否在运行、工作流 id、状态、当前步骤等。 |
| `/clear` | 清除当前会话（与 `reset_session` 一致）：该会话的历史与工作流状态会被清空。 |

指令按**整条消息为指令或“指令 + 仅空白”**匹配。例如 `/stop` 会匹配，`/stop 现在` 不会。

## 创建工作流

### 通过描述

发送以 `/workflow` 开头、后面跟任务描述的消息：

```
/workflow 总结仓库中 top 3 的 issue 并起草一份简短状态报告
```

机器人会生成工作流定义（步骤）、执行并在每步后持久化状态。

### 使用自己的 JSON

可以自行提供 JSON 工作流定义，使用围栏代码块（如 ` ```json `）或单条消息中 `/workflow` 后的 `{ ... }` 对象。示例：

**消息内容：** 先输入 `/workflow`，换行后接 JSON 块：

```json
{
  "id": "report-1",
  "name": "周报",
  "steps": [
    { "id": "fetch", "type": "llm", "description": "获取最近 7 天活动" },
    { "id": "ask_title", "type": "user_input", "description": "输入报告标题", "input_key": "title" },
    { "id": "write", "type": "llm", "description": "根据标题和活动撰写报告" }
  ]
}
```

机器人会解析、展示步骤列表并用任意语言请你确认；确认后开始执行。

**步骤类型**：

- **`llm`**：一次模型调用完成该步的 `description`。不使用工具，输出作为该步结果保存。
- **`user_input`**：机器人发送该步的 `description` 作为提示并等待你的回复，回复会存入 `input_key`（此类型必填）。

## 恢复与控制

- **user_input** 步骤之后，在同一会话中直接回复即可，该条消息会作为工作流输入。
- 若在**超时**（默认 30 分钟）内未回复，工作流会暂停，之后可用 `/workflow continue` 或 `/resume` 恢复。
- 当有任务**正在运行**时，你发送的非控制指令会收到简短提示，列出：`/stop`、`/status`、`/clear`、`/resume`。
- **`/stop`** 会取消当前运行（工作流或其他任务）。工作流状态会以已取消形式保存，你仍可查看或通过 `/clear` 清空会话。
- **`/status`** 输出当前会话及工作流状态（如 `workflow_id`、`status`、`current_step_index`）。
- **`/clear`** 会清除该会话的对话与工作流状态，相当于重新开始。

## 配置

在 Synbot 配置中可设置：

- **`workflow.userInputTimeoutSecs`**（默认：`1800`）：`user_input` 步骤等待用户输入的最长时间（秒），超时后暂停（如 30 分钟）。
- **`workflow.workflowsRoot`**（可选）：工作流状态持久化目录。未设置时使用 `config_dir/workflows`（如 `~/.synbot/workflows/`）。

状态按会话存储（每个 session key 一个工作流状态文件）。每步完成后都会持久化，因此恢复时会从上一未完成步骤继续。

## 速查

| 操作 | 方式 |
|------|------|
| 用描述创建 | `/workflow <你的任务描述>` |
| 用 JSON 创建 | `/workflow` + JSON 块；被询问时确认 |
| 暂停/超时后继续 | `/workflow continue` 或 `/resume` |
| 停止当前运行 | `/stop` |
| 查看会话与工作流状态 | `/status` |
| 清空会话与工作流 | `/clear` |

所有与工作流相关的消息都会写入该会话的对话历史，以便在步骤间和恢复时保留上下文。
